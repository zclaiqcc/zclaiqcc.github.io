Code Review

目的：

  • 通过翻看之前的业务代码，一方面解决之前遗留的问题，另一方面熟悉业务逻辑

形式：

  • 临时采用跨模块方式进行，每周至少对两个模块进行 Code Review，并做好相应的记录（发现的问题以及可能的解决方案）

要求：

  1. 不需要全面熟知业务，直接从代码逻辑看，如果感觉有优化的空间，再结合业务去分析是否可以进行优化
  2. 以 10k 群为考虑依据，分析有没有明显的低性能代码，标记并尝试给出解决方案
  3. 对于之前不熟悉的地方以及发现的新知识，记得做笔记，尝试给其他人的代码加注释

Weekly Share

目标：

  • 核心是通过技术分享，提供一个分享和交流的机会，先富带动后富，实现共同富裕

形式：

  • 内部每周固定时间（暂定周三下午 17 点）进行技术分享，时长 30 分钟，提问交流 30 分钟，并确定下周分享主题

要求：

  1. 需要前期准备，有书面形式的内容（文档、脑图等）可以分享会后发送给大家
  2. 思路清晰，最好能提前列一个大纲
  3. 难度不要太高，目标是简单易懂
  4. 每次附带一个相关的小任务给感兴趣的人一个上手尝试的方式

部分问题的解决方案

关于技术给出问题解决时间的预期的解决方案

  1. 接到问题后，半小时以内定位问题，商务同事可以在有技术接单的情况下统一回复客户半小时给预期进度
  2. 如果很难定位或者需要更多的时间，说明并给出其他合理时间点
  3. 如果需要调整优先级则相关人员临时拉会讨论，技术给出方案以及可能产生的影响（部分功能暂时不可用、延期等）
  4. 问题分为 一小时内、半天、一天、一天以上
  5. 执行人员不对最后问题的解决时间延期负责，但是技术整体对这个时间负责，总是延误可能会导致技术整体利益受损

简化在于

  1. 不需要再考虑问题规模给出定位问题的时间（默认半小时）
  2. 不直接对解决问题的延期负责，可以有更充分的时间考虑相对合理的方案，需要支持可以叫团队其他人一起帮忙
